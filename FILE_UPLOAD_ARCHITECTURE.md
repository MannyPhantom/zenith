## Key Design Decisions

### 1. **Two-Layer Storage**
- **Files** â†’ Supabase Storage (binary data)
- **Metadata** â†’ PostgreSQL (searchable info)
- Linked via `file_url` field

### 2. **Smart Abstraction**
- `project-data-supabase.ts` handles Supabase detection
- Falls back to mock data if not configured
- No code changes needed to switch

### 3. **Real-Time Updates**
- Events notify components of changes
- Cache invalidation keeps data fresh
- Optimistic UI updates for speed

### 4. **Progressive Enhancement**
- Works with mock data (no Supabase)
- Works with database only (no storage)
- Full features with both configured

### 5. **User-Friendly Interface**
- Drag-and-drop for easy uploads
- Progress indicators for feedback
- Confirmation dialogs for safety
- Loading states for clarity

## Performance Optimizations

1. **Client-Side Caching**
   - 5-second cache for repeated requests
   - Reduces database queries

2. **Optimistic Updates**
   - UI updates immediately
   - Background sync with server

3. **Efficient File Handling**
   - Direct Supabase URLs (no proxying)
   - Streaming for large files
   - Parallel uploads for multiple files

4. **Smart Re-Rendering**
   - Only affected components update
   - Event-driven architecture
   - Local state management

## Scalability Considerations

- **Storage**: Supabase Storage scales automatically
- **Database**: PostgreSQL handles millions of records
- **Organization**: Files grouped by project
- **Cleanup**: Cascade deletes remove orphaned files
- **Bandwidth**: CDN-backed storage for fast delivery
